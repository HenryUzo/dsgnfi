Developer Guideline

How to prepare a React template for plug-and-play CMS import

This guideline is for frontend developers building a React project that should become a CMS template after import. The goal is not to upload a finished website and run it directly. The goal is to structure the frontend so the CMS can recognize each page, recognize each section on each page, and expose those sections as editable content inside the CMS page editor.

Developer goal
Build once, import cleanly, edit in CMS
Import target
Structured pages and editable sections
Core requirement
Sections must be declared, serializable, and mappable

The main principle

A CMS-importable React template must be treated as structured content, not just layout. Every page must have a known identity, and every editable section must have a stable type, stable data shape, and predictable location in the page tree. If the importer cannot identify those things, the template will not be plug and play.

Bad fit for import Good fit for import
Page assembled from deeply nested custom components with no clear section boundaries. Page composed from named, top-level sections with explicit section types.
Text and media hardcoded directly across many files. Content passed through serializable section props or section data objects.
Dynamic route generation with hidden path logic. Explicit route declarations or a manifest that lists page routes.
One-off animation wrappers with no reusable section meaning. Presentation layer separated from section content structure.

Required project structure

The React project should be prepared so the importer can find the app root, routes, assets, and page sections without reverse-engineering the entire codebase.

template-project/ package.json template.dsgnfi.json src/ App.tsx pages/ HomePage.tsx AboutPage.tsx ContactPage.tsx sections/ HeroSection.tsx FeaturesSection.tsx GallerySection.tsx FaqSection.tsx data/ home.sections.ts about.sections.ts public/ assets/ hero.jpg gallery-1.jpg intro-video.mp4

Required

  • `package.json`
  • `src/App.tsx`
  • page entry components
  • `public/assets` for importable media
  • `template.dsgnfi.json` manifest

Recommended

  • one `pages/` directory
  • one `sections/` directory
  • separate `data/` files per page
  • clear section naming conventions

Avoid

  • only shipping `dist/`
  • burying routes inside factories
  • mixing content and runtime fetch logic everywhere
  • opaque layout-only wrapper trees

The manifest contract

A plug-and-play import should not depend only on regex over `App.tsx`. The project should provide a manifest that tells the CMS what pages exist, which sections appear on those pages, and which section schema each one should map to.

{ "name": "Agency Editorial Template", "framework": "react-vite", "category": "agency", "assetsDir": "public/assets", "routes": [ { "path": "/", "pageKey": "home", "component": "HomePage" }, { "path": "/about", "pageKey": "about", "component": "AboutPage" }, { "path": "/contact", "pageKey": "contact", "component": "ContactPage" } ], "pages": [ { "pageKey": "home", "title": "Home", "slug": "/", "sections": [ { "id": "home-hero", "type": "hero", "label": "Hero", "component": "HeroSection", "dataSource": "src/data/home.sections.ts#hero" }, { "id": "home-features", "type": "features", "label": "Features", "component": "FeaturesSection", "dataSource": "src/data/home.sections.ts#features" } ] } ] }

Manifest rules

  • Every page must have a stable `pageKey`.
  • Every section must have a stable `id` and `type`.
  • Every section should point to a single data source.
  • Section `type` must map to a known CMS block type.

What the CMS gets from this

  • recognizable pages
  • predictable section ordering
  • block editor selection by section type
  • safe asset import and rewriting

How page sections should be built

Each section should behave like a CMS block. That means the section needs a clear identity and a serializable content shape separate from its presentation logic.

One section = one block type
A section component should map to a single conceptual CMS block. If one component mixes hero, testimonials, gallery, and CTA in one blob, the importer cannot expose them cleanly in the editor.
Serializable content only
Section data should be plain JSON-compatible values: strings, numbers, booleans, arrays, objects, and asset references. Avoid hiding content inside closures, hooks, or imperative DOM mutations.
Separate content from effects
Motion, transitions, cursor effects, and interaction behavior can exist in the component, but the editable content model must be separate. The CMS edits the section data, not the animation code.
Stable field names
If a hero uses `title`, `subtitle`, `imageUrl`, and `cta`, keep those names stable. The CMS editor depends on stable field names to show correct editing controls.
Explicit section ordering
A page should declare its sections in order. The CMS should not have to infer order by crawling arbitrary JSX trees.

Recommended React coding pattern

The easiest import target is a page built from a sections array plus presentational components.

export const homeSections = [ { id: "home-hero", type: "hero", data: { title: "Design systems for modern brands", subtitle: "A clearer path from story to launch", imageUrl: "/assets/hero.jpg" } }, { id: "home-features", type: "features", data: { heading: "What we do", items: [ { title: "Strategy", description: "Find the right direction" }, { title: "Design", description: "Shape the experience" } ] } } ]; export function HomePage() { return <Page sections={homeSections} />; }

Why this works well

  • The importer can read section identity directly.
  • Each section already looks like a CMS block payload.
  • Data can be validated before import.
  • The CMS page editor can reuse the same field structure.

What to avoid

  • building a page from many tiny anonymous fragments
  • scattering page copy across five files with no master shape
  • deriving content from side effects instead of explicit data
  • making the importer infer semantics from CSS classes

How the CMS should recognize sections

For a section to be editable in the CMS page editor, the system needs a direct mapping between the imported section type and a known editor schema.

React template provides CMS must resolve
`type: "hero"` editor form for hero fields and public renderer for hero blocks
`type: "features"` repeatable item editor plus public features section renderer
`type: "gallery"` media picker/editor plus gallery renderer
new custom section type new block schema, new editor UI, and new public renderer

Plug and play only works when imported section types already have editor support. If a developer invents a new section type, the CMS team still has to implement that block type before import can be considered complete.

Developer checklist before upload

Project readiness

  1. Use React + Vite.
  2. Keep page routes explicit.
  3. Provide `template.dsgnfi.json`.
  4. Store media in `public/assets`.
  5. Keep one canonical data source per page.

Section readiness

  1. Give each section a stable `id`.
  2. Give each section a stable `type`.
  3. Keep section data serializable.
  4. Map each section type to a known CMS block.
  5. Do not hide editable content inside animation wrappers.

Import pipeline the team should implement

Step 1

Validate the project

Confirm framework, manifest presence, route contract, and asset directory. Reject malformed bundles with precise errors.

Step 2

Read pages and sections

Build a normalized page model from the manifest and section data instead of reverse-engineering arbitrary JSX.

Step 3

Map to CMS blocks

Convert recognized section types into CMS-native block payloads and attach asset URLs rewritten into CMS storage.

Step 4

Publish template

Save supported pages, navigation defaults, and settings defaults as a custom template version.

Step 5

Materialize into a site

Create missing pages for the selected site and preserve already modified pages unless the operator chooses to replace them.

Step 6

Edit from the CMS

Each recognized section should appear in the page editor with the correct form fields for its block type.

Definition of plug and play

Condition Expected result
Developer uploads a valid React template bundle Importer accepts it without manual code edits.
Page manifest declares routes and sections CMS creates recognizable pages in the correct order.
Section types map to known CMS blocks Sections are editable in the CMS page editor.
New section type appears with no CMS support Import should warn or fail until block support exists.
Site already has modified pages Importer preserves modified pages unless overwrite is explicitly requested.

The correct mental model is: React source defines the template intent, the manifest defines the import contract, and the CMS stores the final editable page/block data.
